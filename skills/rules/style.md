# Style Rules

## Purpose
These rules keep the repository consistent, readable, and easy to review by enforcing clear naming and documentation habits.

## Writing Expectations
- Use descriptive, domain-specific names that communicate intent.
- Prefer concise wording that makes the purpose of a symbol, section, or note obvious at a glance.
- Avoid ambiguous or temporary identifiers that hide meaning.

## Examples
- Preferred: `PlayerWeaponInventoryHealth`
- Avoid: `tempvar1`, `var2`

## Common Violations
- Using vague names that do not describe the object or role.
- Reusing generic identifiers across unrelated contexts.
- Writing notes that are technically correct but too terse to be useful to a reviewer.

## Best Practices
- Prefer names that reflect the domain concept and behavior.
- Keep naming consistent across similar files and symbols.
- Use short, direct prose when the content is intended for later reuse or handoff.

## Validation
- Review the naming and phrasing in new or modified content for clarity, consistency, and usefulness.
