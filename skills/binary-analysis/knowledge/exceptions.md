# Exception Handling Metadata

## Overview

Exception handling metadata in PE binaries supports structured unwinding, stack frame recovery, and runtime exception dispatch. On Windows x64, the runtime function table is essential for correct stack walking and exception delivery. On x86, Structured Exception Handling (SEH) uses stack-based registration records.

Exception metadata is also a valuable analysis artifact, because it often defines function boundaries and reveals compiler-generated exception handling constructs.

## Windows x64 exception model

### Runtime function table

The PE exception directory points to an array of `RUNTIME_FUNCTION` entries, one per function or function range. Each entry includes:
- `BeginAddress`
- `EndAddress`
- `UnwindInfoAddress`

The loader or debugger uses this table during unwinding and when exceptions are raised.

### Unwind info

`UNWIND_INFO` describes how to restore registers and stack state. It includes:
- prolog length
- unwind codes
- frame register and offset
- exception handler information

The unwind codes encode operations like stack allocation, register saves, and chained unwind info.

## x86 SEH model

### Exception registration records

On x86, SEH handler chains are linked through registration records on the stack. Each record contains a pointer to the next record and a pointer to the exception handler.

### Exception dispatch

When an exception occurs, the OS walks the SEH chain and invokes handlers until one returns `EXCEPTION_CONTINUE_EXECUTION` or `EXCEPTION_EXECUTE_HANDLER`.

### Compiler-generated frames

C++ exceptions and structured exception frames are built using compiler-generated prolog/epilog sequences and registration records. The stack may contain frame pointers, object destructors, and catch/filter metadata.

## Analysis workflow

### 1. Locate exception metadata

In PE, inspect the data directory for `IMAGE_DIRECTORY_ENTRY_EXCEPTION`. For x64, parse the runtime function table. For x86, identify SEH registration records and exception dispatcher thunks.

### 2. Use unwind metadata for function recovery

Unwind info often maps code ranges to function boundaries and can recover valid call stacks even in stripped binaries.

### 3. Identify exception-based obfuscation

Protections and malware often use exceptions for control flow or anti-debugging. Look for:
- deliberate page faults and exception handlers
- disguised jump tables inside exception filters
- stack unwinding used as anti-analysis state transitions

### 4. Validate unwind info

Check whether unwind entries cover the entire code range and whether chained unwind info points to valid tables.

## Practical diagnostics

- Broken or missing unwind data can prevent debuggers from displaying meaningful call stacks.
- A function with no unwind info on x64 is treated as non-unwindable; this can hide call frames in crash dumps.
- Corrupt `UNWIND_INFO` is a common symptom of custom loader unpacking or obfuscated code regions.

## Developer and reverse engineering notes

- Exception tables are generated by the compiler and linker; they usually match function boundaries precisely.
- Windows x64 requires unwind metadata for all non-leaf functions. Leaf functions may omit unwind info if they do not modify the stack.
- On x86, SEH registration records are often present only when structured exception handling is used or when compilers emit frame pointers.

## Example patterns

### x64 runtime function entry

- `BeginAddress` and `EndAddress` are RVAs within the code section.
- `UnwindInfoAddress` points to unwind metadata that may include chained entries.

### x86 SEH record chain

- Each stack record points to the next registration block.
- The handler is usually a compiler-generated frame unwind or landing pad.

## Common mistakes

- Treating exception metadata as optional on x64; if it is missing, unwinding fails.
- Assuming page faults always indicate bugs; in obfuscated binaries, exceptions can be a control-flow mechanism.
- Ignoring chained unwind entries that may lead to recursive metadata.

## Practical Guidance

- Use unwind metadata to recover accurate call stacks during crash analysis; if unwind info is missing, combine static heuristics with runtime traces.
- Be aware that exception-based control flow is often used by protectors; instrument and capture faults to reveal payload behavior.

## Tools & Commands

- WinDbg: `!analyze -v`, `!exchain`, and `!unwind` to inspect exception chains and unwind info.
- `eu-unwind` / `llvm-objdump` / custom scripts to parse runtime function tables in stripped images.

## Validation Checklist

- Confirm unwind entries cover expected address ranges and that chained unwind info points to valid metadata.
- Validate that SEH chains on x86 are intact and that exception handlers do not point to non-code regions.
