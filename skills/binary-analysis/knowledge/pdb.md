# Program Database (PDB)

## Overview

Program Database (PDB) files store debugging symbols, type information, and source mapping for Windows binaries. They connect compiled code to high-level symbols, source files, line numbers, and variable metadata.

PDB files are generated by the Microsoft toolchain and are often separated from the shipped binary. Recovering or matching PDB data can dramatically speed reverse engineering and crash analysis.

## PDB architecture

A PDB uses a multi-stream file (MSF) format. The file header points to a stream directory, and each stream contains a distinct category of data.

### Stream types

Common PDB streams include:
- stream 1: free page map
- stream 2: directory of symbol streams
- stream 3: section contributions and module info
- symbol streams: public and private symbols
- type info streams: type records and ID lists
- line number and source file streams

### Signature and age

A PDB contains a GUID and age value. The binary’s debug directory stores the same signature and age to ensure an exact match.

### Public vs private PDBs

- Public PDBs contain a limited symbol set, typically export names and public function symbols.
- Private PDBs contain full type information, local variables, source file paths, and complete debug symbols.

## Debug directories in PE

The PE debug directory points to debug data and may contain:
- CodeView (`RSDS`) information for PDB lookup
- older `NB10` debug formats
- FPO and exception information for legacy debugging

The `CodeView` record contains the PDB path, GUID, age, and PDB filename.

## Analysis workflow

### 1. Identify PDB metadata

Inspect the PE debug directory and locate `IMAGE_DEBUG_DIRECTORY` entries. Read the `CodeView` signature to extract the PDB GUID, age, and filename.

### 2. Verify the PDB match

Confirm the PDB’s GUID and age match the binary’s debug record. If they differ, symbol loading can fail or produce incorrect mappings.

### 3. Load symbols

Use APIs such as `SymInitialize`, `SymLoadModuleEx`, and `DbgHelp` or the DIA SDK to load symbols from the PDB.

### 4. Extract symbol and type data

From the PDB, recover:
- function names and public symbols
- source file paths and line numbers
- type records for classes, structs, enums, and function signatures
- local variable and argument names

### 5. Handle missing or partial PDBs

If the PDB is missing, use:
- export and import data
- RTTI and vtables
- heuristic symbol recovery
- public symbol servers and symbol caches

## Practical examples

### Load PDB symbols using DbgHelp

```cpp
#include <windows.h>
#include <dbghelp.h>

void LoadPdbSymbols(HANDLE process, const char* image_path) {
    SymInitialize(process, nullptr, FALSE);
    SymLoadModuleEx(process, nullptr, image_path, nullptr, 0, 0, nullptr, 0);
}
```

### Use DIA SDK for advanced queries

The DIA SDK offers programmatic access to PDB streams, symbol records, type information, and source line mappings.

## Reverse engineering notes

- PDBs are often stripped from shipped products, but symbols may still be published separately or available through symbol servers.
- Partial PDBs are common in release builds; they may expose only public symbols and exports.
- Incorrect symbol matching is a frequent source of analysis errors.

## Debugging notes

- Symbol loading failures commonly result from mismatched PDB GUID or age.
- Public symbols may be sufficient for stack traces but insufficient for local variable reconstruction.
- Use symbol server redirection to resolve known binaries when the PDB path is old or unavailable.

## Common mistakes

- Loading a PDB by filename alone without checking GUID and age.
- Assuming the original PDB path on disk is valid for a different machine.
- Attempting to use a public PDB for full source-level debugging when only limited symbols exist.

- DIA SDK (`msdia140.dll`) for programmatic PDB access and symbol extraction.

## Validation Checklist

- Confirm the PDB GUID and age match the binary's debug directory before using the PDB for symbol resolution.
- Verify loaded symbols produce expected stack traces and source mappings in a controlled debug session.
